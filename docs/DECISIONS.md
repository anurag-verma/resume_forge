# Decisions Log

Decisions made during implementation where the planning docs didn't specify an exact choice. Simplest option that keeps the app free and client-side wins.

## RB-001 — Scaffold project

- **create-vite's default template scaffolds React 19 + oxlint.** Pinned `react`/`react-dom` to `^18.3.1` (Architecture doc §2 fixes React 18) and swapped oxlint for ESLint (flat config, `typescript-eslint`) + Prettier, matching Architecture doc §2's stated lint/format tools.
- **Vitest version.** Started with vitest `^2.1.8` (paired with Vite 5 internally) but the project pins Vite `^6`, which produced duplicate/incompatible nested `vite` installs and a TS type-conflict in `vite.config.ts`. Bumped to vitest `^3.0.4`, which supports Vite 6 natively — resolves cleanly with no duplicate `vite` in `npm ls`.
- **`tsconfig.app.json` / `tsconfig.node.json`:** removed `erasableSyntaxOnly` (requires TypeScript 5.8+; project pins `~5.7.2` per nothing more modern being required yet) to avoid an unknown-compiler-option error. Re-added `@types/node` (needed for `tsconfig.node.json`'s `"types": ["node"]" and vite.config.ts).
- **Vitest config location:** used `import { defineConfig } from 'vitest/config'` in `vite.config.ts` (test config merged into the main Vite config) rather than a separate `vitest.config.ts`, per Architecture doc's single `vite.config.ts` in the project tree.

## RB-002 — Data model & types

- **Entry discriminated union.** Architecture doc §4 only spells out `ExperienceEntry` as an "e.g." example of the `Entry` union. Extended the same pattern to the rest: `SummaryEntry`, `EducationEntry`, `SkillEntry`, `ProjectEntry`, `CertificationEntry`, `LanguageEntry`, `CustomEntry` — each carries a literal `type` field (matching the entry's `SectionType`) so the union discriminates cleanly, and each got the minimal fields its section needs (e.g. `SkillEntry` has an optional `group` for RB-015's grouping toggle; `ProjectEntry` has optional `technologies: string[]`).
- **`createId()` helper.** Thin wrapper around `nanoid()` in `src/lib/id.ts`, per Architecture doc §2's nanoid choice for section/entry IDs.

## RB-003 — Zustand stores with persistence

- **Two localStorage keys, not one.** Architecture doc §5.1 names a single key (`resume-builder-data`) but §3's file tree splits state into `useResumeStore` and `useSettingsStore`. Since `AppData.settings` is genuinely separate data from `AppData.resumes`, persisted each store under its own key (`resume-builder-data`, `resume-builder-settings`) rather than forcing both zustand stores to share one key. This also isolates corruption: a corrupt settings blob doesn't take resume content down with it, and vice versa.
- **Debounce implemented in the storage adapter, not via a wrapper action.** `src/lib/persistStorage.ts` debounces at the `localStorage.setItem` call itself (per-key `setTimeout`), so every store that uses it gets the ~500ms debounce from Architecture doc §5.1 for free, and `getItem` handles corrupt-JSON recovery (backs up the raw string to `${name}-corrupt`, returns `null` so the store falls back to its default state) and swallows `QuotaExceededError` per Security doc §5 / Architecture doc §7, rather than duplicating that logic in each store.
- **Migration scaffold kept minimal.** `migrate: (persistedState) => persistedState as PersistedState` — a real migration function only makes sense once `SCHEMA_VERSION` actually increments past 1; this is the scaffold point future tickets hook into.
- **Cross-store cleanup.** `useResumeStore.deleteResume` calls `useSettingsStore.getState().removeSettings(id)` and `duplicateResume` copies the source resume's settings to the new id, so per-resume settings don't silently orphan or default unexpectedly when resumes are deleted/duplicated.

## RB-010 — App shell & layout

- **Tailwind's default breakpoints line up exactly with Frontend spec §2's** (mobile < 768, tablet 768–1023 via `md:`, desktop ≥ 1024 via `lg:`), so no custom `screens` config was needed in `tailwind.config.js`.
- **Design tokens as real CSS custom properties.** Frontend spec §1.1 names the tokens literally as `--ink`, `--paper`, etc., so they're defined in `src/styles/index.css` `:root` and Tailwind's `colors` config references them via `var(--ink)` rather than hardcoding hex values twice. Also added `--desk` (`#EDECE7`, spec §2.1's preview-pane background) alongside the documented token table since it's used the same way.
- **Toolbar/EditorPanel/PreviewPane content is a placeholder shell only.** This ticket is layout/chrome; the resume switcher shows the active resume's real name (already available from `useResumeStore` — free correctness, no extra work) but its dropdown, and every toolbar action (Template, Customize, Import/Export, Download PDF, Save status), are static/inert pending their own tickets (RB-024, RB-025, RB-031/032, RB-030, RB-041). The A4 page frame and zoom controls are deliberately left out of `PreviewPane` — those belong to RB-020.
- **Verification tooling:** no `chromium-cli` available in this environment; used a local `npx playwright` + Chromium install (scratchpad-only, not committed) to screenshot the app at 1440px/820px/390px and confirm the layout against spec §2 at all three breakpoints, catching a toolbar-wrapping bug at tablet width that unit tests wouldn't have caught.

## RB-011 — Personal Info form

- **New shared `ui/TextField.tsx`.** Ticket calls for 8 similar labeled inputs; extracted a small labeled-input component (label/error/hint wiring, `aria-describedby`/`aria-invalid`) now rather than duplicating the same markup 8 times — this is immediate reuse within the ticket, not speculative, and later section forms (RB-013/014) will reuse it too.
- **Email validation is soft per the ticket's AC.** `PersonalInfoForm` always calls `updatePersonalInfo` on every keystroke regardless of validity; an invalid email only renders a `role="alert"` warning underneath (via `TextField`'s `error` prop) and never blocks typing or reverts the value.

## RB-012 — Section accordion framework

- **Confirm dialog built on native `<dialog>`, not a hand-rolled modal.** `ui/ConfirmDialog.tsx` uses `showModal()`/`close()` — the browser gives focus trapping, `Escape`-to-cancel, and a backdrop for free, which is exactly what Frontend spec §3.8/§6 asks for (destructive-action confirmation, focus trap, Esc) without adding a modal library or writing custom trap logic. jsdom 25 doesn't implement `showModal`/`close` yet, so `src/test/setup.ts` polyfills both with a minimal open-attribute toggle — test-only, doesn't ship in the production bundle.
- **Add-section menu uses `role="menu"`/`role="menuitem"`**, not plain buttons in a `<div>`, so the dropdown options are unambiguous from a section card's own inline-editable title button (which is also a `<button>` with the same visible text, e.g. two things both named "Custom Section" on screen at once). This is also the more correct ARIA pattern for a popup menu.
- **Section body content is still a placeholder** ("This section's fields will appear here.") — the actual per-type entry forms are RB-013/014/015's job. This ticket only covers the accordion chrome: inline-editable title, visibility toggle, collapse, add-menu, and remove-with-confirm.
- **New section defaults to visible, with entries: []; title seeded from `SECTION_TYPE_LABELS`.** Renaming afterward is free-text via the inline editor — no special-casing needed since `Section.title` was always separate from `Section.type` in the data model (RB-002).

## RB-013 — Experience section form

- **`SectionCard` now takes a `children` prop instead of hardcoding its body.** RB-012 had the placeholder text baked into `SectionCard` itself; that would have made it reach into per-type form components (`ExperienceForm`, and later Education/Skills/etc.), an awkward dependency from a generic "shared" component into type-specific ones. Instead `SectionCard` just renders `children` (falling back to the old placeholder text when none is given), and a new `SectionBody.tsx` in `editor/` — not `editor/shared/` — owns the `section.type` switch and is passed in by `SectionList` as the child. `shared/` stays a one-way dependency (types import from it, it imports nothing type-specific).
- **Added `duplicateEntry` to `useResumeStore`.** RB-003 only shipped `addEntry`/`updateEntry`/`removeEntry`/`reorderEntries`; RB-013's AC explicitly requires entry duplication, which no existing action covered. Mirrors `duplicateResume`'s pattern: clone with a fresh `createId()`, insert immediately after the original (not appended at the end), so duplicated experience entries stay next to their source.
- **`DateRangeInput` (not just a bare month/year picker) is the shared component**, combining the start-date picker, end-date picker, and "Present" checkbox as one unit — Architecture doc §3 names `DateRangeInput` explicitly under `shared/`, and Education (RB-014) needs the same start/end/"ongoing" pattern, just with different checkbox copy (`presentLabel` prop) — e.g. "Currently studying here" instead of "Currently working here".
- **Month/year stored and edited as plain `YYYY-MM` strings**, per Frontend spec §3.3's explicit month+year `<select>` pair (avoids locale/format parsing bugs). `lib/date.ts` centralizes the format/parse/range helpers so Education and any future dated entry type reuse the same logic instead of re-deriving it.
- **Unchecking "Currently working here" defaults the end date to the current year-month** (via `currentYearMonth()`) rather than leaving it blank — keeps `endDate` always a valid `YYYY-MM` once "Present" is off, and the user adjusts the pre-filled value rather than picking from two "unset" selects.
- **Markdown-lite toolbar (B/I/bullet) is out of scope here** — the Description field is a plain `<textarea>` for now; the constrained-markdown renderer and toolbar are RB-016's job specifically because of its own security requirements (Security doc §4.1, whitelist-only rendering).

## RB-014 — Education, Projects, Certifications, Languages, Summary forms

- **`CustomSectionForm` was deliberately left out**, even though Architecture doc §3 lists `CustomSectionForm.tsx` as a planned file and users can already add multiple "Custom Section"s via RB-012's add-section menu. No ticket in doc 05 schedules building it (RB-014's title/scope names Education/Projects/Certifications/Languages only), so a custom section still falls through to `SectionBody`'s placeholder text. Flagging here rather than silently building unrequested scope — worth a follow-up ticket.
- **Extended `ui/TextField` with an optional `onBlur`**, and added `ui/SelectField` (mirrors `TextField`'s API for `<select>`) — both are now reused across Projects (technologies commit-on-blur), Certifications, and Languages (proficiency), not one-off inline markup per form.
- **Projects' `technologies` list is a plain comma-separated text field, not a full tag/pill input.** RB-015 is specifically scoped to build the Enter-to-add tag input UX for Skills; building that same interaction here for Projects would preempt RB-015's deliverable. `TechnologiesField` keeps its own local draft string (`useState`) and only commits the parsed `string[]` to the store `onBlur`, so retyping mid-edit doesn't fight a controlled re-render on every keystroke (a raw split-on-every-keystroke approach eats trailing commas as you type).
- **Certifications use a single `MonthYearPicker`, not `DateRangeInput`** — `CertificationEntry.date` is one point in time, not a range, per the data model (RB-002).
- **Language proficiency is a curated `SelectField`** (Native/Fluent/Professional/Conversational/Basic), not free text — keeps output consistent for a value that's inherently a small closed set, matching the app's "curated, not freeform" design direction (Frontend spec §1).
- **`SummaryForm` auto-creates its one `SummaryEntry` in a `useEffect`** rather than requiring the user (or `addSection`) to seed it — `Section.entries` is generically `Entry[]` for every type, so an empty summary section starts with `entries: []`; the effect fills in the single entry the first time the form renders, and the entry-card add/duplicate/remove pattern doesn't apply here (ticket explicitly calls for "single textarea").

## RB-015 — Skills tag input

- **"Drag to reorder" (Frontend spec §3.4) is implemented as ‹ › move-earlier/move-later buttons, not actual pointer drag-and-drop.** RB-017 ("Drag-and-drop reordering") is its own dedicated ticket that adds dnd-kit with a keyboard sensor for both sections and entries; building real drag interactions here would duplicate that work ahead of schedule. The buttons give equivalent (keyboard-accessible, in fact more accessible than raw drag) reordering now; RB-017 can later swap the mechanism without touching the underlying `reorderEntries` store action, which both approaches share.
- **"Skills persist and render in preview" — the persistence half is fully verified** (store + component tests); the *preview* half can't be fully verified yet because `PreviewPane` is still RB-020/021's placeholder — there is no template rendering pipeline to check skills against. Skills flow through the same `useResumeStore` the (future) preview will read from, so no rework is anticipated once RB-020/021 land.
- **Grouping is derived UI state, not a new persisted field.** `Section` has no `groupingEnabled` flag — the "Group skills" checkbox is local `useState`, defaulting to `true` only if any existing skill already has a `group` set (so reopening a resume with grouped skills shows them grouped). This avoids a schema/migration change for what the ticket marks as an optional P1 sub-task; the underlying `SkillEntry.group` field already existed from RB-002.
- **Reordering within a group required care:** a naive "swap with the next id in the full entries array" breaks when the adjacent array element belongs to a different group. `SkillsForm.moveEntry` instead reorders just the visible subset (all skills, or one group's skills) and re-slots those ids back into their original positions in the full array — verified by a dedicated test (`reordering within one group does not affect another group`).
- **Group rename edits every member entry's `group` field**, not a separate group entity — there's no group data structure independent of the strings on each `SkillEntry`, so renaming "Languages" to "Programming Languages" loops over matching entries and calls `updateEntry` on each.

## RB-016 — Markdown-lite descriptions

- **The renderer builds React elements directly (`React.createElement`), never an HTML string.** `lib/markdownLite.tsx` only ever constructs `<strong>`, `<em>`, `<ul>`, `<li>`, `<br>`, and `Fragment` — there is no `dangerouslySetInnerHTML` anywhere, and no step that parses user text as markup. All raw user text is passed as React children, which React escapes as text nodes automatically. This is a stronger guarantee than "sanitize the HTML string with DOMPurify then inject it": there is no HTML string to sanitize in the first place, so a bug in a sanitizer's allowlist can't reopen an XSS hole. Security doc §4.1's "no `dangerouslySetInnerHTML` with user input" and "whitelist parser" requirements are both satisfied by construction, not by a runtime check.
- **No link syntax at all.** The markdown-lite spec supports only `**bold**`, `*italic*`/`_italic_`, and `- `/`* ` bullet lines — Security doc §4.1 specifically calls out "no links with `javascript:` URIs"; the simplest way to guarantee that is to never parse anything into an `<a>` in the first place. A literal `[text](javascript:...)` just renders as inert plain text (verified by test).
- **Unmatched markers fall back to literal text** (e.g. `this ** is not bold` renders unchanged) rather than throwing or guessing intent — keeps the parser simple and never produces surprising partial markup.
- **`MarkdownLiteTextarea` (the B/I/• toolbar) only touches the three Description fields already built in RB-013/014** (Experience, Education, Projects) — Summary's textarea and any future Custom Section field were out of this ticket's stated scope ("Markdown-lite **descriptions**"); swapping them to the same component later is a one-line change per field since the component is already shared.
- **Toolbar actions read `textarea.selectionStart`/`selectionEnd` directly off the DOM ref**, not React state — text selection isn't something React tracks in a controlled input's value, so the wrap/bullet handlers reach through the ref at click-time, mutate the string, call `onChange`, then restore focus and cursor position via `requestAnimationFrame` (needed because the DOM hasn't repainted the new value yet when the click handler runs).

## RB-017 — Drag-and-drop reordering

- **One shared `SortableList` component wraps `DndContext`/`SortableContext` for every reorderable list** (sections, and entries in each of the 5 `EntryCard`-based forms) instead of duplicating the dnd-kit boilerplate 6 times. It takes `items`, `onReorder(orderedIds)`, and a render-prop `children`, so `SectionList` and each entry form only need to supply their list + the matching store action (`reorderSections` / `reorderEntries`).
- **The id-reordering math is a separate pure function, `lib/reorder.ts#computeReorderedIds`**, unit-tested directly (6 tests: forward, backward, drop-on-self, unknown active/over id). Actual pointer/keyboard drag gestures are notoriously unreliable to simulate in jsdom (collision detection depends on real layout measurements jsdom doesn't compute), so the correctness-critical logic is isolated and tested in complete isolation from any DOM/drag simulation.
- **Mouse + keyboard reordering, and persistence, were verified live in a real browser (Playwright), not in jsdom.** A jsdom unit test can render the drag handle and confirm it's focusable, but can't faithfully simulate an actual pointer drag past dnd-kit's activation-distance threshold or its `closestCenter` collision detection. Did a real mouse-drag (section list and an entry list), a real keyboard sequence (Tab to the handle, Space to pick up, Arrow to move, Space to drop), and a page reload in between to confirm the reordered state survives the debounced localStorage write — all three passed with zero console errors.
- **`GripVertical` on both `SectionCard` and `EntryCard` changed from a decorative icon to a real focusable `<button aria-label="Drag to reorder …">`** carrying dnd-kit's `attributes`/`listeners` — needed for the keyboard sensor to have something tabbable to activate, and incidentally makes the drag handle itself screen-reader-discoverable (it wasn't before).
- **`EntryCard` gained a required `id` prop** (previously it had no notion of its own id — the parent only used `key={entry.id}`, which React never exposes to the component). `useSortable({ id })` needs that id explicitly.
- **Skills tags keep their RB-015 arrow-button reordering, not dnd-kit.** `SkillsForm`'s pills are a flex-wrap layout, not a vertical list — `verticalListSortingStrategy` doesn't fit them, and a `rectSortingStrategy` variant would be a separate effort. RB-015's AC (reordering works) is already satisfied by the arrow buttons; revisiting skills specifically for pointer-drag tag reordering is deferred rather than bundled into this ticket.

## RB-020 — Preview pane with A4 page frame

- **On-screen pagination is a visual approximation, not real layout pagination.** The actual exported PDF is paginated by the browser's print engine via `@page` CSS in `window.print()` (RB-030, per Architecture doc §5.3) — that's the source of truth for where pages really break. `PagedDocument` renders resume content as one continuous A4-width column and overlays dashed lines + "Page N of M" labels at every multiple of 297mm of *measured* content height. This matches Frontend spec §3.6's literal wording ("dashed line indicator where page breaks fall") without trying to reflow content into physically separate, gapped page boxes — which would require either a real pagination engine (e.g. paged.js) or manually re-slicing DOM content per page, both out of scope for an MVP preview indicator.
- **The pagination math is a separate pure function, `lib/pagination.ts#computePagination`**, unit-tested directly (5 tests) with a controlled/mocked height — exactly the RB-017 pattern (`computeReorderedIds`) applied again, because real height measurement can't be trusted in jsdom (it doesn't do layout) but the math determining page count/break offsets from a given height absolutely can and should be tested precisely.
- **Real height measurement (`ResizeObserver` + `scrollHeight`) was verified in an actual browser, not jsdom.** jsdom has no `ResizeObserver` at all (polyfilled as a no-op stub in `src/test/setup.ts` purely so components don't throw) and returns 0 for `scrollHeight` on arbitrary content. So the jsdom tests mock `scrollHeight` directly to prove the pagination *math* renders correctly, while a live Playwright check injected 40 real paragraph elements into the actual rendered DOM and confirmed the browser's real `ResizeObserver` fired, `scrollHeight` reflected the true layout height, and the component correctly showed "Page 1 of 2" / "Page 2 of 2" with one break line — closing the gap jsdom can't cover.
- **No real resume template exists yet** (that's RB-021+) — `PreviewPane` accepts an optional `children` prop and falls back to the existing placeholder text when none is given. Once RB-021 wires in `ResumeDocument`, it becomes `PreviewPane`'s child and the pagination/zoom mechanics apply to it automatically with no changes needed here.
- **Zoom state unifies the three discrete levels (`'fit' | 75 | 100`) with continuous Ctrl/Cmd+scroll adjustment** (Frontend spec §3.6) by typing zoom as `'fit' | number`: the buttons set exact values, while wheel+modifier nudges a numeric percentage (clamped 25–200% via `lib/zoom.ts#clampZoomPercent`, its own 3 unit tests) and switches out of `'fit'` mode. Frontend spec only mentions three fixed buttons *and* separately mentions scroll-zoom as an interaction nicety (§3.6, §5); reconciling them as one numeric-or-fit state avoids needing two disconnected zoom mechanisms.
- **"Fit" means fit-to-width**, computed from the scroll container's measured `clientWidth` via `ResizeObserver`, matching Frontend spec §2.2's "zoom defaults to fit-width" (stated for tablet, applied here as the sensible default at any width since there's no separate "fit-height" mode called for).

## RB-021 — Classic template

- **No template registry yet.** Architecture doc §5.2 describes a shared registry (`templates/index.ts`) that all templates plug into, but RB-024 ("Template gallery & registry") is the dedicated ticket for that — including the gallery modal, thumbnails, and `React.lazy` code-splitting. Building a registry now with only one real entry (and Modern/Minimal as placeholder no-ops) would be dead weight until those templates exist. `ResumeDocument` renders `ClassicTemplate` directly; switching it to a real registry lookup when RB-022/023/024 land is a small, contained change.
- **CSS variables are the actual mechanism, not just computed inline values.** `lib/templateStyles.ts#getTemplateCssVars` turns `Settings` into real custom properties (`--accent`, `--font-heading`, `--font-body`, `--line-height`, `--section-spacing`, `--font-size-base`) set once on the template's root `style`, and every heading/rule/paragraph inside references `var(--x)` rather than receiving a computed prop. This is what Architecture doc §5.2 means by "templates use CSS variables driven by Settings" — a future Customize panel (RB-025) changing `accentColor` just updates one variable, with no re-render of the template's structure needed.
- **A minimal font-pair registry (`lib/fontPairs.ts`) with exactly one entry (`classic-serif`: Source Serif 4 / Source Sans 3)** was added now rather than waiting for RB-025's full curated font-pair picker — Classic's spec (Frontend spec §4) names its fonts explicitly, so the template needs *some* concrete mapping from `Settings.fontPairId` to real font-family strings today. Falls back to the same pair for any unrecognized id. No font files are self-hosted yet (that's RB-043); family names rely on system fallback stacks in the meantime, consistent with the RB-010 decision to defer font-file hosting.
- **`renderMarkdownLite` output needed a wrapper to restore bullet markers.** RB-016's parser deliberately emits bare `<ul>`/`<li>` with no className (whitelist-only, no attributes). Tailwind's base reset strips `list-style` from all `<ul>` by default, so without intervention bullets silently disappeared in the rendered resume — caught during the live browser check, not by unit tests (jsdom doesn't apply Tailwind's compiled CSS during component tests, so the missing bullets weren't visible there). Fixed with one shared `MarkdownLiteBlock` wrapper (Tailwind arbitrary-child selectors: `[&_ul]:list-disc [&_ul]:pl-5`) used everywhere a description renders, rather than patching the security-sensitive parser itself.
- **"Headings never orphan at page bottom" is satisfied via CSS (`break-after-avoid-page` on h2/h3, `break-inside-avoid` on entry wrappers), verified by class-presence tests**, not by observing an actual page break — these properties are inert during RB-020's continuous on-screen preview and only take real effect once the browser paginates for print (RB-030's `@page` CSS). Testing "did this orphan get prevented" requires an actual paginated rendering context that doesn't exist until RB-030; testing "is the correct CSS property applied to the correct element" is the right-sized check available now.
- **The "full sample resume" test fixture is a hand-built `Resume` object covering every section/entry type** (not RB-040's `sampleData.ts`, which doesn't exist yet — that's a P2 Epic 4 ticket for onboarding/empty-states, not a prerequisite for template correctness). Includes a hidden section to confirm `visible: false` sections are excluded from template output.

## RB-022 — Modern template

- **Extracted `SectionEntries`/`EntryHeading`/`EntryMeta`/`MarkdownLiteBlock` out of `ClassicTemplate.tsx` into `templates/shared/SectionEntries.tsx`** the moment a second template needed the identical per-entry-type field rendering (~170 lines). RB-021 kept this logic local to Classic since there was only one consumer then; Modern needing byte-for-byte the same "what does an Experience/Skills/etc. entry say" logic (differing only in surrounding layout) is exactly the second-use case that justifies pulling it out now rather than duplicating or prematurely abstracting during RB-021.
- **Shared test fixture (`testFixtures.ts`)** replaces what was a resume object duplicated inline in `ClassicTemplate.test.tsx`. Both templates now render off the identical `fullTestResume`, which also directly backs the "lossless switching" AC: a dedicated cross-template test renders the same resume through both `ClassicTemplate` and `ModernTemplate` and asserts every key piece of content (name, each entry's heading text, etc.) appears in both outputs.
- **Sidebar membership is fixed by section *type*** (`skills`, `languages` → sidebar; everything else → main column), independent of the section's position in `resume.sections`. Frontend spec §4 describes Modern's sidebar as containing "skills/contact/languages" as a structural trait of the template, not a reflection of user-chosen section order — so `ModernTemplate` buckets by type and preserves each bucket's own relative order, rather than trying to read positional intent (e.g. "sections placed after some marker go to the sidebar") out of the flat section list.
- **Added an `inter` font pair** (heading = body = Inter) to `lib/fontPairs.ts` so Modern's spec-recommended default look ("Inter throughout") is demonstrable — Settings still drives font choice generically via `getTemplateCssVars`, exactly like Classic; `ModernTemplate` does not hardcode Inter. There is no UI yet to actually switch `templateId`/`fontPairId` (that's RB-024/025), so verifying Modern's look live required temporarily flipping `DEFAULT_SETTINGS` for a screenshot and reverting immediately after — not a permanent change, and confirmed via the full test suite passing both before and after the revert.
- **Sidebar tint uses `color-mix(in srgb, var(--accent) 8%, white)`** rather than computing an rgba/hex tint manually from the arbitrary accent hex string in JS — simpler, and safe for the evergreen-browser target (Architecture doc §5.5; `color-mix` has been supported in all major engines since 2023).

## RB-023 — Minimal template

- **"Accent only on name underline" is taken literally: section (h2) and entry (h3) headings get zero accent styling** — no color, no border — unlike Classic (accent rules throughout) or Modern (accent heading color + sidebar tint). A dedicated test asserts every `h2` has an empty inline `color`/`borderColor`, specifically to guard against accidentally reintroducing accent styling by copy-pasting from Classic/Modern (an easy mistake since all three templates share the same `SectionEntries` rendering).
- **Added a `spectral-inter` font pair** (Spectral heading / Inter body) to `lib/fontPairs.ts`, following the same pattern as RB-021's `classic-serif` and RB-022's `inter` — one concrete pair per template's spec-recommended default look, added incrementally as each template needs to demonstrate its own typography rather than front-loading a full curated set ahead of RB-025.
- **Consolidated the cross-template "lossless switching" test into its own file, `losslessSwitching.test.tsx`**, parameterized over all three templates (`it.each`) instead of leaving it bolted onto `ModernTemplate.test.tsx` (where it only made sense back when Modern was the second template). Now that a third template exists, one shared, table-driven test is clearer than duplicating the same content-presence assertions per template-pair file.
- **`ResumeDocument` became a `switch` instead of an `if`** now that there are three template ids to route — still a plain conditional, not RB-024's registry; same reasoning as RB-021/022 for why the registry itself waits.
- **Wide margins implemented as `20mm` page padding** (vs Classic/Modern's `12mm`) — both still comfortably clear the "safe margins ≥ 12mm" floor from Frontend spec §4's shared template rules, while visibly delivering on Minimal's "wide margins... elegant whitespace" description. Section spacing also gets a `1.5×` multiplier (`calc(var(--section-spacing) * 1.5)`) on top of the user's chosen spacing setting, for the same whitespace-forward effect without introducing a second spacing setting.

## RB-024 — Template gallery & registry

- **Native `<dialog>`-based `Modal` almost broke the "only active template chunk loads" AC.** `<dialog>` toggles *visibility* via `showModal()`/`close()` — it does not unmount its children. Since `Toolbar` always renders `<TemplateGalleryModal>` (just with `open={false}` initially), and the gallery's thumbnails each mount a `React.lazy` template component, all three template chunks were fetching immediately on page load regardless of the modal ever being opened — caught only by the live Playwright network check, not by any unit test (jsdom doesn't fetch real chunks). Fixed by gating `Modal`'s children on `{open && children}`, so content — and anything lazy inside it — only mounts once the dialog is actually shown. This is a correctness fix to the shared `Modal` component, not a one-off workaround in the gallery.
- **Verified code-splitting two ways:** the production build output itself (separate `ClassicTemplate-*.js`/`ModernTemplate-*.js`/`MinimalTemplate-*.js`/`SectionEntries-*.js` chunks — Vite automatically deduplicated the shared `SectionEntries` module into its own chunk rather than triplicating it), and a live `vite preview` + Playwright network-log check confirming exactly one template chunk (`ClassicTemplate`) loads on initial page load, with `ModernTemplate`/`MinimalTemplate` only fetching once the gallery modal is opened.

## RB-030 — PDF export via print CSS

- **Print hiding uses Tailwind's built-in `print:` variant directly in JSX (`print:hidden` on the app shell, `hidden print:block` on a dedicated print-only `<ResumeDocument>` render), not a hand-rolled `visibility: hidden` CSS rule in `print.css`.** Tailwind ships the `print:` media-query variant with zero config, so `print.css` only needs the one thing Tailwind genuinely can't express — the `@page { size: A4; margin: 0 }` at-rule. This keeps the print-specific logic colocated with the component tree it affects instead of split across a CSS file that has to be kept in sync with markup changes.
- **A separate, unscaled `<ResumeDocument>` renders specifically for print**, rather than trying to print `PreviewPane`'s on-screen version. The on-screen preview carries zoom scaling, the desk background, and `PagedDocument`'s dashed page-break/`"Page N of M"` overlay (RB-020) — none of which belong in the actual exported PDF, which should show only the clean template output at 100% scale. Rendering the same `ResumeDocument`/registry twice (screen + print) doesn't double-fetch any `React.lazy` template chunk, since the dynamic `import()` promise is cached per module regardless of how many call sites render the resulting component.
- **`document.title` swap for the suggested filename, restored via the `afterprint` event** (fires whether the user saves or cancels) rather than leaving the title changed — there's no other cross-browser JS hook to influence a "Save as PDF" destination's suggested filename, and mutating `document.title` is the established technique other print-to-PDF tools rely on.
- **Filename sanitization had to preserve spaces and hyphens, not strip everything unsafe in one pass.** The regex needed care: stripping spaces *before* splitting the name into first/last parts would collapse "Jane Doe" into one word ("JaneDoe") instead of "Jane-Doe" — caught immediately by `buildResumeFilename`'s own unit tests, which is exactly the kind of subtle off-by-one-step bug a test written alongside the implementation is meant to catch before it reaches a live check.
- **First-use tooltip is a distinct step the user must click through ("Continue to print"), not a hint that fires alongside `window.print()`.** A native print dialog is typically a blocking, page-covering UI; showing our own tooltip at the same moment it opens would mean the user never actually gets to read it. Seen-state is a plain `localStorage` flag (`lib/pdfTooltip.ts`) — this is ephemeral one-time UI guidance, not resume data, so it doesn't need to go through the Zustand-persisted stores.
- **Cross-browser rendering (Chrome/Firefox/Edge/Safari) could only be verified for Chromium in this environment.** Verified: real DOM text (zero `<canvas>` elements) in correct top-to-bottom reading order, app chrome fully hidden under `@media print`, and the exact `visibility`-independent "print only this element" pattern used here is a long-established, broadly-supported CSS technique — but Firefox/Safari/Edge weren't independently exercised. Flagging as a manual QA step before shipping, same as RB-010's tablet-toolbar check would have needed real-device coverage this environment can't provide.

## RB-031 — JSON export/backup

- **Two distinct export shapes, both discriminated by a `kind` field not present on the base `AppData` type (RB-002).** `AppDataExport` (`kind: 'app-data'`) backs up every resume + all settings; `SingleResumeExport` (`kind: 'single-resume'`) is just one resume + its own settings, for sharing/transferring a single resume without the rest of the user's library. `AppData` itself only describes what's persisted to localStorage — the export *file* format is deliberately a superset so RB-032's importer can reliably tell which shape it's looking at without guessing from structure alone.
- **Export logic reads directly from the Zustand stores' `getState()`** (`useResumeStore.getState()`, `useSettingsStore.getState()`) rather than requiring the caller to pass resume/settings data down as props through every layer — `buildAppDataExport()` takes no arguments at all. `buildSingleResumeExport(resume, settings)` still takes explicit arguments since it exports a specific resume the caller already has in hand (e.g. from a component's own store subscription), not "whatever is currently active."
- **"Re-importable" was verified for this ticket as "the exported shape survives a JSON round-trip losslessly and is self-describing enough to import"** — actual import (file picker, schema validation, hostile-input rejection) is RB-032's dedicated ticket. Each export builder has a round-trip test: `JSON.parse(JSON.stringify(export))` must deep-equal the original, proving no data is lost in serialization before RB-032 ever has to parse it back.
- **A real end-to-end download was verified live** (Playwright with `acceptDownloads: true`), not just that `downloadJsonFile`'s internals were called correctly in jsdom — confirmed an actual file lands on disk, is valid parseable JSON, and matches the expected `kind`/`schemaVersion`/content shape. jsdom can't trigger a real browser download, so this was the one part of the AC that needed a live check rather than a unit test.
- **A TypeScript error only `tsc -b` (the production build) caught, not `vitest run`:** a test's `vi.fn(() => 'blob:mock-url')` had no inferred parameter type, so `mock.calls[0][0]` indexing failed type-checking even though the test ran and passed fine under Vitest's esbuild-transpile-only execution. Fixed by giving the mock an explicit call signature (`vi.fn<(blob: Blob) => string>(...)`) — a reminder that `npm test` passing doesn't guarantee `npm run build` will, since Vitest doesn't type-check by default.

## RB-032 — JSON import with validation

- **Hand-rolled schema validators, not a validation library (zod, etc.).** Same reasoning as RB-016's hand-rolled markdown-lite parser: Security doc §4.3 says minimize dependencies (each one is attack surface), and the schema is fully known and finite (one `Resume`/`Settings`/8 `Entry` variants) — a hand-written validator is auditable line-by-line and adds zero new supply-chain surface. Every validator reads only the named fields it expects off the raw input and constructs a *brand-new* object from them; nothing is ever spread from the untrusted input, so unknown/extra keys (Security doc §4.1's "strip unknown keys") are absent by construction, not by an explicit deny-list step that could be forgotten for a new field.
- **Rejection is strict, not lenient/partial.** Any single malformed field, entry, or section fails the *entire* import with a descriptive error, rather than silently dropping just the bad piece and importing the rest. This is safer and more predictable than partial recovery — a user should never end up with a resume that's silently missing a section because one entry inside it was malformed.
- **`readAndValidateImportFile` checks file size before ever reading file contents**, and JSON.parse is wrapped in its own try/catch distinct from the schema-validation step — each failure mode (empty, oversized, unparsable, wrong shape) gets its own specific, user-facing error message rather than one generic "import failed."
- **`file.text()` (the modern, correct, standard Blob API) doesn't exist in jsdom.** Rather than downgrading production code to a more roundabout API for the sake of the test environment, added a `File.prototype.text` polyfill (backed by `FileReader`, which jsdom does implement) to `src/test/setup.ts` — the same pattern already used for `<dialog>` and `ResizeObserver` gaps in earlier tickets.
- **Explicit hostile-input tests**, not just "invalid input" tests: a `__proto__`-injection payload (verifies `Object.prototype` is untouched and the app doesn't crash) and a payload with a string value containing a code-injection-shaped string (verifies nothing ever gets `eval`'d or otherwise executed — we only ever call `JSON.parse`, never `eval`/`Function`, per Security doc §4.4). Also verified live in a real browser: a hostile file with malformed `personalInfo` and `settings` produces a clear "Import failed" dialog with zero native `alert()` dialogs firing (i.e., nothing executed).
- **Merge always regenerates every id** (resume, every section, every entry) via `withFreshIds`, for both single-resume and app-data imports — imported content can never collide with an existing resume's ids, which matters since ids are also the join key into `useSettingsStore`'s per-resume settings map.
- **Replace has different semantics for single-resume vs. app-data imports, deliberately.** Single-resume replace overwrites *only the currently active resume's* content while keeping its existing id (so it doesn't orphan any other stored reference to that id); app-data replace wholesale discards every existing resume and setting, since a full-backup restore is meant to be a total reset by definition. Both are still explicit, user-confirmed choices via `ImportConfirmDialog` — nothing is ever auto-applied without the merge/replace prompt.

## RB-041 — Save status indicator

- **A new non-persisted `useSaveStatusStore` (status/`lastSavedAt`/`pendingCount`) tracks real debounced-persistence state**, replacing the RB-010-era static "✓ Saved" placeholder chip that never actually reflected anything. `persistStorage.ts`'s `setItem`/`removeItem` call `beginSave()`/`endSave(success)` directly — the debounce mechanism itself is the single source of truth for save status, rather than having each store re-derive or duplicate it.
- **`pendingCount`, not a plain boolean, because two independent stores debounce independently.** `useResumeStore` and `useSettingsStore` each have their own debounce timer under their own localStorage key; if the chip flipped to "saved" the instant either one's timer fired, it would misreport "saved" while the other store's write was still pending. Incrementing on the *first* `setItem` call for a given key (not on every rescheduling) and decrementing once that key's write actually lands keeps the count accurate even though both stores share one global chip.
- **A failed write (quota exceeded) still clears the "saving" state but does not stamp `lastSavedAt`.** `endSave(false)` intentionally omits the timestamp bump — RB-034's quota-exceeded toast is the surface that tells the user something went wrong; the chip's job here is narrower ("does this reflect debounced persistence accurately"), so it just avoids lying that a save completed when it didn't, without inventing a third visual state the ticket never asked for.
- **Caught and fixed a double-checkmark bug during the live visual check**: the lucide `Check` icon plus the ticket's own "✓" character in the label text rendered as two checkmarks side by side ("✓ Saved ✓ 21:47") — invisible to the unit tests (which only assert on text content, not the rendered icon), only visible in an actual screenshot. Removed the icon; the "✓" is baked into the label text alone now, matching Frontend spec §3.1's literal "Saved ✓" wording.
- **`aria-live="polite"` plus `role="status"`** on the chip (Frontend spec §5's "live region announces 'Saved' politely") — the visible label text (including the timestamp) is what gets announced, which is a superset of the literal word "Saved" rather than a narrower announcement, since a screen-reader user benefits from hearing the time too.
